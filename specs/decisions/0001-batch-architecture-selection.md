# ADR 0001: 배치 아키텍처 선택

## 상태
승인됨 (Accepted)

## 배경
일일 1,000만 건의 레코드를 처리하는 정산 시스템이 필요합니다. 2시간 이내의 짧은 배치 윈도우 내에서 높은 트랜잭션 무결성과 성능이 요구됩니다.

## 제안된 아키텍처
**로컬 파티셔닝(Local Partitioning) 기반의 Spring Batch**

### 결정 이유
- **성능:** 사용자 제공 지표(500만 건 / 8분 55초)에 따르면, 최적화된 Spring Batch는 1,000만 건 목표를 20분 이내에 충분히 처리할 수 있음을 입증했습니다.
- **복잡도:** 현재의 1,000만 건 규모에서 Apache Spark나 원격 파티셔닝(Remote Partitioning)에 비해 운영 복잡도가 낮습니다.
- **일관성:** 금융 정산에 필수적인 강력한 RDBMS 트랜잭션 지원(ACID)을 활용할 수 있습니다.

### 확장 로드맵
1. **Phase 1 (10M):** 단일 노드 Spring Batch (m5.2xlarge) + 로컬 파티셔닝.
2. **Phase 2 (50M+):** 일일 처리량이 단일 노드의 임계치(vCPU/IOPS)를 초과할 경우, 멀티 노드 원격 파티셔닝 또는 Apache Spark 하이브리드 구조로 전환 검토.

## 성능 참조 지표
- **기준점 (사용자):** 5,000,000 건 처리 / **8분 55초** (최적화 모드)
- **프로젝트 목표:** 10,000,000 건 처리 / **20분 이내**

## 최종 결정
로컬 파티셔닝을 사용한 Spring Batch를 채택하며, `CursorItemReader`와 `JdbcBatchItemWriter`를 핵심 최적화 전략으로 사용합니다.
