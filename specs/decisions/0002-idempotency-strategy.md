# ADR 0002: 정산 멱등성 보장 전략

## 상태
승인됨 (Accepted)

## 배경
금융 정산 시스템은 어떠한 경우에도 중복 지급이나 이중 계상이 발생해서는 안 됩니다. 멀티 스레드 배치 환경에서 멱등성을 보장하기 위한 3중 방어 전략이 필요합니다.

## 결정 사항: 3중 방어 전략

### 1단계: 물리적 제약 - 복합 유니크 인덱스 (DB 레이어)
DB 레벨에서 물리적으로 중복 삽입이 불가능하도록 설정합니다.
- **인덱스 구성:** `정산일자` + `사용자ID` + `서비스유형` + `트랜잭션ID`.
- **효과:** 중복 삽입 시도 시 즉시 제약 조건 위반 에러를 발생시켜 해당 청크를 롤백합니다.

### 2단계: 구현 전략 - Upsert 패턴 (코드 레이어)
실패한 배치를 재실행할 때, 이미 성공한 파티션을 수동으로 정리하지 않고도 안전하게 재시도할 수 있게 합니다.
- **수행 내용:** `INSERT INTO ... ON CONFLICT DO NOTHING` (PostgreSQL) 또는 `ON DUPLICATE KEY UPDATE` (MySQL) 사용.
- **결과:** 이미 처리된 레코드는 무시하고(Skip), 처리되지 않은 레코드만 안전하게 정산합니다.

### 3단계: 로직 검증 - 상태 기반 낙관적 잠금 (비즈니스 레이어)
원본 트랜잭션 레코드의 상태값을 활용하여 단 한 번만 처리되도록 보장합니다.
- **SQL 조건:** `UPDATE source_transactions SET status = 'COMPLETED' WHERE id = ? AND status = 'READY'`.
- **수행 내용:** 업데이트된 행의 수가 정확히 1일 때만 정산 결과 레코드를 생성합니다.

## 결정 근거
RDBMS의 ACID 속성과 유니크 인덱스 결합은 금융 데이터의 무결성을 보장하는 업계 표준입니다. Redis와 같은 캐시는 속도는 빠르나, 영속성과 트랜잭션 측면에서 정산 시스템의 요구사항을 완벽히 충족하기 어렵습니다.

## 고려된 대안
- **Redis Bloom Filter:** 속도는 빠르나 미세한 오탐 확률(False Positive)과 캐시 휘발 시 데이터 유실 위험이 있음.
- **사전 조회 후 삽입 (Select-before-Insert):** 고도의 병렬 환경에서 레이스 컨디션에 취약하며, 이를 해결하기 위한 테이블 락은 성능 저하를 초래함.
